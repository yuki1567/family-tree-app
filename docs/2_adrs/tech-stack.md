# 技術スタック

家系図アプリ FamilyTree で採用する技術スタックと、その選定理由をまとめたドキュメント。

- ステータス: 事前選定は完了（写真の保存先・ホスティングはデプロイ時に決定する）
- 関連 issue: #2
- 関連ドキュメント: [要件定義書](../1_requirements/requirements.md)

## 背景
技術スタックの選定は、プロジェクト開始前に固める唯一の事前決定事項とし、その他の設計は各機能の実装時に行う方針とする。本ドキュメントには、決定したレイヤとその理由・却下した代替案を記録していく。

選定にあたって踏まえる要件の要点:
- Web アプリ / クラウド保存 / 認証必須 / レスポンシブ（PC・スマホ）
- 対応ブラウザは Chrome・Safari の最新版
- 数千〜数万人規模のツリーを破綻なく描画する
- 写真はサーバ側に保存する
- CSV のインポート／エクスポート
- 環境変数はクラウド側で管理する（`.env` は使用しない）

## 決定事項

### 1. 言語／アーキテクチャの背骨 — TypeScript フルスタック
フロントエンド・バックエンドとも TypeScript を採用する。

**前提**
- ブラウザで動くフロントエンドは、プラットフォーム上の制約から実質 JS/TS 一択である。これは選択肢ではなく与件。

**理由（＝バックエンドも TypeScript に揃える理由）**
- フロントが JS/TS で確定している以上、バックエンドも TypeScript に統一すれば、言語・ツールチェーン・思考の切り替えが1つで済む。ソロ＋エージェント開発では構成がシンプルなほど有利。
- `Person`／`Relationship` などの型をフロント・バックで共有でき、複雑な関係（再婚・養子・異母兄弟・不明な人物）の不整合バグを減らせる。
- Claude 主体の開発のため、ドキュメント・実例が豊富で正確なコードを書きやすい TypeScript/JS エコシステムが適する。

**却下した代替案**
- フロントは TS・バックは別言語（Go / Python / Rust 等）: 性能や学習目的では有力だが、型共有ができず、ソロ＋エージェント開発では構成が重くなるため見送り。

### 2. アーキテクチャの形 — SPA ＋ 独立バックエンド API
フロントエンド（SPA）とバックエンド API を分離し、モノレポで型を共有する。

**理由**
- 将来的に外部サービス連携を行う可能性があり、独立した API があれば API ファーストで供給しやすい。
- 業務ロジック（関係の整合性・CSV マージ・認可など）をバックエンドに集約でき、フロント／バックの境界が明確になる。
- モノレポの共有パッケージで `Person`／`Relationship` 型を共有でき、決定 1 の型共有方針を維持できる。

**却下した代替案**
- フルスタック・メタフレームワーク（Next.js / Nuxt で画面と API を 1 アプリに）: 可動部品が少なく立ち上がりは速いが、UI と API が 1 デプロイに密結合し、将来 API を外部クライアントへ供給する段階で作り直しが必要になるため見送り。

**留意点（コストとして受け入れる）**
- 2 デプロイ・CORS・クロスオリジン認証・型共有のビルド配線など、構成の複雑さが増える点は許容する。

### 3. フロントエンド — React + Vite（メタフレームレス SPA）
React をライブラリとして使い、ビルドツールに Vite を用いたクライアントサイド SPA とする。メタフレームワーク（Next.js 等）は採用しない。あわせて用いる主要ライブラリ:
- ルーティング: React Router
- サーバ状態・データ取得: TanStack Query
- （フォーム処理などは実装時に必要に応じて追加）

**理由**
- 決定 2（独立バックエンド API）と最も素直に噛み合う。データは自分の API から HTTP で取得し、TanStack Query がキャッシュ・再取得・楽観的更新を担う（大規模ツリーの順次読み込み FR-V03、検索 FR-P04、モーダル編集 FR-V05 に有効）。
- 全画面ログイン必須で SEO・公開ページが不要なため、SSR／メタフレームワークの主目的が効かない。
- 部品が少なく、フロントは静的ホスティングのみでデプロイできる。

**却下した代替案**
- Next.js（フロントとして利用）: 市場価値が高く将来的に学ぶ価値はあるが、SEO・チーム標準化の利点が個人・非公開・ソロでは効かず、独立 API 構成では SSR／API ルートの使いどころも減る。

### 4. バックエンド（API フレームワーク）— Hono
独立した API サーバのフレームワークに Hono を採用する。

**理由**
- 軽量・モダンで、TypeScript の型推論が強く、構成をシンプルに保てる。
- RPC により API の型をそのままクライアントへ共有でき、決定 1（型共有）・決定 2（独立 API）の方針と噛み合う。
- 多ランタイム対応で、将来のホスティング選択の幅も広い。

**却下した代替案**
- Fastify: 高速・実績豊富だが、Hono の RPC のような端から端までの型共有は標準では得にくい。
- NestJS: DI・モジュールで構造化できるが、この規模・ソロ開発には重く、抱える概念が多い。
- Express: 情報量は最大だが設計が古く、型サポートが弱い。

### 5. データベース — PostgreSQL（ORM: Drizzle）
データストアに PostgreSQL を、データアクセス層に Drizzle ORM を採用する。データは人物と「夫婦・パートナー関係」「親子関係」を扱う関係性中心のもので、整合性が重要、クラウド保存・複数端末アクセスが前提。

**理由（PostgreSQL）**
- 関係中心のデータで整合性が重要。リレーショナルの制約（外部キー・一意制約など）により「子は必ず夫婦・パートナー関係の下」等のルールを DB レベルで守れる。
- 祖先/子孫の世代たどり（FR-V02／FR-V03）を再帰 CTE で素直に書ける。数千〜数万人規模なら十分な性能。
- クラウド保存・複数端末アクセス（要件）に合うサーバ型 DB で、マネージドの選択肢が豊富。
- JSON 型も持つため、`備考` など柔軟性が欲しい箇所はドキュメント的にも扱える。

**理由（Drizzle）**
- TypeScript ファーストで型が強く、決定 1（型共有）・決定 4（Hono／RPC）の型重視の方針と揃う。
- SQL に近い記述で生成 SQL が見えやすく、再帰 CTE 等も書きやすい。SQL の感覚が身につく（学習面の利点）。
- 軽量で Hono との相性が良い。

**却下した代替案**
- MySQL: 要件は満たすが、再帰 CTE・制約・JSON 周りで PostgreSQL の方が素直。
- SQLite: 軽量だが、クラウド・複数端末前提ではサーバ型が自然（Turso 等はあるが今回の前提と外れ気味）。
- ドキュメント DB（MongoDB / Firestore）: 関係がグラフ状の多対多のため、参照＋アプリ側 join になり整合性も自前管理。世代たどりが N クエリ化しやすく、本ドメインと噛み合わない。
- グラフ DB（Neo4j 等）: ドメイン的には最も自然だが、運用が重く TS エコシステムが手薄、Cypher が学習の脇道、数千〜数万規模では過剰。PostgreSQL の再帰 CTE で同等の探索を軽く実現できるため見送り。
- Prisma（ORM）: スキーマの明快さ・移行ツール・Prisma Studio など DX は優れる。ただし「重さ」がある:
  - コード生成（`prisma generate`）を伴い、従来は Rust 製クエリエンジンのバイナリ同梱でバンドル／コールドスタートが重く、エッジ等のランタイム制約も出やすい（近年は driver adapters で軽量化が進行中だが、決定 4 の Hono 多ランタイム方針とは噛み合わせが落ちる）。
  - 再帰 CTE のような複雑クエリは Prisma Client API では書けず `$queryRaw`（生 SQL）に落ちるため、本 app の世代たどり（FR-V03）では型付き API の利点が効きにくい。
  - SQL が隠れがちで学習面でも不利。
  - 以上より、軽量で純 TS・SQL に近い Drizzle を優先する。

### 6. バックエンドの実行モデル — 常駐サーバ/コンテナ型（サーバレスは不採用）
バックエンド API は**常駐プロセス（Node サーバ）**として動かす。具体的なホスティング先は引き続きデプロイ時に決定するが、「常駐サーバ/コンテナ型」か「サーバレス（FaaS）型」かの方針はここで固定する。

**理由**
- 関係中心データの整合性と祖先/子孫の再帰 CTE 探索（FR-V02／FR-V03）を、常駐プロセスのコネクションプールで安定して扱える。サーバレスは多数の短命実行が PostgreSQL の接続を食い潰し、RDS Proxy / PgBouncer 等のプーラが実質必須になって複雑さが増す。
- 現状の `@hono/node-server`（常駐 Node サーバ）のまま進められ、開発環境（コンテナで Node サーバを常駐起動）と本番の実行モデルが一致する。
- CSV インポート・写真アップロード等を、FaaS のペイロード/実行時間制限（6MB 同期・15 分等）に縛られず素直に扱える。
- Hono は多ランタイム対応（決定 4）のため、Cloud Run / App Runner / Fly / Railway / ECS / EC2 など「コンテナや Node が常駐できる環境」に広く載せられる。具体ホストは budget 等に応じてデプロイ時に選ぶ。

**却下した代替案**
- サーバレス（Lambda 等 FaaS）: アイドル時ゼロスケールで安価だが、(1) Postgres 接続管理にプーラ前提の追加設計が要る、(2) コールドスタート、(3) `hono/aws-lambda` 等へのアダプタ差し替えで dev（常駐）と prod（関数）が乖離、(4) CSV/写真がペイロード・時間制限に当たりやすい。本アプリの関係データ＋再帰 CTE＋ファイル処理の性質に対して難所が多く、学習の本筋からも逸れるため見送り。

**留意点**
- これは実行モデルの方針決定であり、具体的なホスティング先（Cloud Run / ECS / Fly 等）は引き続きデプロイ時に決定する（下記参照）。

## デプロイ時に決定する項目（本選定の対象外）
以下はアプリのコード構造を縛らず、デプロイ段階の選択・設定で差し替えられるため、事前のスタック選定には含めない。各機能の実装／デプロイ時に決める。
- **写真の保存先**: オブジェクトストレージ（S3 互換 API）を抽象化して扱えば、具体的なプロバイダ（S3 / Cloudflare R2 等）はデプロイ時の選択・設定で差し替えられる。
- **ホスティング先（フロント／API／DB）**: フロントは静的ホスティング、API は Hono が**常駐できるコンテナ/Node 実行環境**（決定 6）、DB はマネージド PostgreSQL（接続文字列で切替）。いずれも実行環境の選択であり、開発開始の前提にはならない。
